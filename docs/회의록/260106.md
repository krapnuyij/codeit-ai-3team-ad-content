## 주요 안건 및 논의사항

### 1. 진행 현황 공유
**각자 진행 상황**:
- 김명환: 파이프라인 수정 및 추가, 서빙 서버 구축
- 김민혁: 텍스트 생성
- 박지윤: 진행 사항 정리, 일정 및 진행 상황 관리, 파이프라인 검토
- 이건희: 휴가
- 이솔형: 이미지 생성

### 2. 핵심 논의사항
**주요 이슈**:
- Step 1 인터페이스 수정
  - 상품 이미지 입력 필수에서 옵션으로 변경. 미입력 시 텍스트(상품 설명+배경 설명)만으로 한 번에 생성
- MCP 서버 일단 제거, 추후 LLM 연동 자동화를 위해 재도입 검토
- 전체 파이프라인 테스트 기준 현제 약 15분, 합성·효과 포함 시 최대 30분
- 기능 우선 구현 후 양자화 예정
- 모델 용량 및 인프라
  - 배경 제거 모델: 소형
  - 이미지 생성 모델: 약 32GB
  - 이미지 합성 모델: 약 32GB
  - 텍스트 생성 모델: 약 5GB
  - 총합 약 70GB, 최대 80~90GB
- 자동 배치
    - 객체와 텍스트 위치 배치를 AI가 상황에 맞게 판단하도록 자동화 필요 (좌표 하드코딩 X, 사람이 일일이 지정 X)
- Qwen 모델
  - 이미지 이해, 위치 판단, 분위기 분석에 강점, 16GB 수준의 비교적 작은 모델, 이미지 분석 성능은 Gemini급 체감
  - 배경 이미지 분석: 분위기 (예: 술집, 해변, 빈티지 테이블), 오브젝트 배치 적합 영역 추천
  - 활용 예: 500원 텍스트를 가장 자연스러운 위치에 배치, 오브젝트를 올릴 수 있는 영역 bounding box 생성
  - 장점: 프롬프트 엔지니어링 부담 감소, Step 1 결과를 Step 3에 연결하는 브릿지 역할
- Flux 모델
  - Flux Dev는 생성에 강점, Flux Pro는 합성에 강점
  - 배경 생성은 Flux Dev, 오브젝트와 텍스트 합성은 Flux Pro 사용
  - 테스트 결과, 이미지 일관성 유지 우수, 프롬프트 오류시 결과 품질 급락하여 조정 필요
- 텍스트를 이미지로 생성
  - 폰트 렌더링 대신 텍스트를 이미지처럼 생성하여 합성
  - 스타일 다양성 증가 및 광고 이미지 자연스러움 향상 기대


**논의된 해결방안**:
- 모델 용량 및 인프라 이슈 해결 방향
    - OS 공용 스토리지(/opt)에 Hugging Face 모델 공유
    - 각자 로컬 HF 캐시는 필요 시 삭제
    - 모델은 읽기 전용 사용, 충돌 없음
    - 용량 99~100% 방지 주의

**팀 결정사항**:
- 중간 통합 방식
  - 완벽하게 만든 뒤 통합하기 보다는 중간 단계라도 인터페이스 맞으면 바로 통합
  - 일정이 촉박하고, 중간 통합을 통해 문제 조기 발견 및 이후 통합 속도 단축 기대
- 테스트 원칙
  - 유닛 테스트(Pytest) 필수, Fail 하나라도 있으면 수정 후 진행
  - Dummy Mode 제공하여 개발자 부담 감소
  - FastAPI 실행, 외부 IP 접근 테스트, Swagger/Docs 확인
