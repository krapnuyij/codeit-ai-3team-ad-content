## 주요 안건 및 논의사항

### 1. 진행 현황 공유
**각자 진행 상황**:
- 김명환: 파이프라인 구조, 서빙 구조 설계
- 김민혁: 이미지 내 텍스트 처리 작업
- 박지윤: 일정 및 진행 사항 관리, 전체 흐름 파악
- 이건희: 프론트엔드 사용자화면에 이미지 업로드, 광고 목적/분위기 설정, 광고 생성 요청 구현
- 이솔형: 이미지 생성 작업 진행

### 2. 핵심 논의사항
**주요 이슈**: 
- 이미지 내 텍스트 처리 방식 논의 
  - 기존 텍스트 제거, 마스크 박스 기반으로 새로운 텍스트 삽입 
  - 현재 임의값으로 테스트 중이나 향후 사용자 직접 입력 또는 AI 추천 방식 모두 가능하도록 설계 예정
  - 텍스트 위치는 AI가 자동으로 적잘한 위치를 탐색하여 삽입, 다양한 이미지에서 추가 검증 필요
- 폰트 처리 전략
  - 전체 폰트 셋은 용량이 커서 테스트용 우선 적용
  - 폰트가 파이프라인에 정상적으로 연동되는지 기술 검증
  - 오픈 라이선스 폰트 사용
- 텍스트 내용
  - 사용자 직접 입력, LLM 자동 추천, 혼합 방식 
- 프롬프트 생성
  - 단일 프롬프트가 아니라 단계별 프롬프트 다수 필요 (배경생성, 텍스트 생성, 텍스트 효과, 합성 등)
- few-shot 활용
  - 프롬프트 엔지니어링 + few-shot 전략 검토
- 서빙 및 비동기 처리 방식
  - 전체 생성 시간 30분
  - 비동기 처리 필수
  - 요청 시 즉시 결과 반환이 아니라 작업시작 + task_id 반환
  - 폴링 방식(클라이언트가 주기적으로 상태 조회)
  - 이벤트/콜백 방식은 서버,네트워크,방화벽 제약 크므로 과제 범위 및 현실적인 구현 측면에서 폴링 방식이 가장 안정적
- 더미 모드
  - 실제 AI 실행 없이 전체 API, 인터페이스, 데이터 흐름 테스트 가능
  - 개발 초기에는 더미모드 ON -> 빠른 개발
  - 통합 테스트 단계에는 더미모드 OFF -> 실제 모델 실행
- Base64
  - 이미지 데이터는 Base64 인코딩 사용
  - 이유: JSON과 충돌 방지, 안정적인 데이터 전송
  - 단점: 데이터 크기 증가
  - 호환성,안정성 측면에서 합리적 선택으로 판단
- 테스트
  - 유닛 테스트
  - 더미 모드 기반 테스트
  - 실제 모델 실행 테스트(no-dummy)

**논의된 해결방안**:

**팀 결정사항**:
- 파이프라인 3단계
  1. 이미지 생성 (누끼, 배경 생성, 합성)
  2. 텍스트 생성
  3. 이미지 + 텍스트 합성
  - 각 단계는 독립 실행 가능하도록 설계
