## 주요 안건 및 논의사항

### 1. 진행 현황 공유
**각자 진행 상황**:
- 김명환: 파이프라인 및 아키텍처 설계
- 김민혁: 휴가
- 박지윤: 일정 및 진행 상황 관리, 파이프라인 및 아키텍처 검토
- 이건희: 프론트엔드, 백엔드 구축
- 이솔형: 이미지 생성 파이프라인 구축 및 테스트 진행

### 2. 핵심 논의사항
**주요 이슈**:
- 전체 아키텍처 설계 및 기본 인터페이스 구축 완료
- 프론트엔드-백엔드-모델서빙 흐름은 REST API 기반으로 연결
- 프로젝트 잔여기간 약 3주 미만으로 이미지 생성,합성,텍스트 효과까지 모든 모델을 다 완성하기엔 리스크가 크므로 전략 재정의 필요
- 이미지 생성 방식 검토
  - 제품 이미지 + 배경 이미지 합성 시 위치 정합, 자연스러움 확보 어려움
  - 이미지 특성 추출을 통해 텍스트 프롬프트 생성하여 이미지 생성시 품질은 좋으나 모델 탐색, 튜닝 비용 큼
  - 텍스트 기반 이미지 생성은 구현은 쉬우나 팀 목표(제품 이미지 기반 광고)와 거리 있음
- Qwen 모델을 사용하여 이미지 분석과 프롬프트 생성 모델로 활용 (이미지 생성 모델 X)
  - 이미지 내 객체, 공간, 분위기, 조명 등 분석
  - 텍스트 넣기 적합한 위치 탐색 가능
  - 한글 입력을 내부적으로 영문 프롬프트로 변환 가능
  - 장점: GPT대비 이미지 공간 이해도 우수, bounding box 추출 가능
  - 활용 가능 시점: 광고 문구를 어디에 배치할지 판단하는 보조엔진, 배경·오브젝트 자동 배치 판단
  - 한계: 모델 자체가 무겁고 양자화·서빙까지 고려해야하므로 필수 엔진이 아닌 선택적 고급 옵션으로 분류
- 프론트엔드/백엔드
  - FastAPI기반, HTML 랜더링 + API 호출 통합 구조
  - 프론트/백엔드 로직 분리, 실제 구현은 단일 서버
  - 프론트: 이미지 업로드, 광고 생성 요청, 상태 조회 및 결과 표시
  - 백엔드: REST API 제공, 모델 서빙 서버 호출, Base64 이미지 송수신
- MCP 관련 논의
  - 필수는 아니라 실험적, 확장 옵션
  - LLM이 API를 직접 호출하도록 돕는 인터페이스 개념
  - 당장 과제 완성의 핵심은 아니므로 가능하면 PoC 수준으로 유지
  - 실패 시 REST API 구조만으로 충분
- 이미지 품질 관련 프롬프트
  - (masterpiece, photorealistic, best quality, cinematic lighting)
  - 이미지 생성 시 품질 편차를 줄이기 위해 고정 품질 프롬프트 적용

**논의된 해결방안**:
- 이미지 생성 시 모든 방식을 동시에 구현하는 것은 무리
  - 제품 이미지는 반드시 입력
  - 그 위에서 합성의 정교함은 단계적으로 조정
  - 2 track으로, 인원 이탈 상황 대비 프롬프트 기반 생성으로 전환 가능한 구조 유지

**팀 결정사항**:
- 한글 텍스트 처리 전략
  - 이미지 생성 단계에서 한글 텍스트를 직접 포함하면 글자 깨짐 발생
  - 특히, Gemini, Nano Banana, 일부 SD 계열 모델에서 심각
  - 따라서 텍스트는 이미지 생성과 분리
  - 1단계: 배경 + 오브젝트 이미지 생성
  - 2단계: 한글 텍스트 전용 모델로 광고 문구 생성
  - 3단계: 텍스트를 이미지 위에 합성
  - 장점: 한글 품질 안정, 폰트·효과 제어 가능, 광고 가독성 확보
- 최종 파이프라인
  - 입력: 제품 이미지 필수, 텍스트 설명(분위기/가격 등) 선택
  - 이미지 처리: 누끼, 배경 생성 또는 단순 배치
  - 텍스트 처리: LLM으로 광고 문구 생성, 폰트/효과 선택 (LLM 추천 가능)
  - 합성: 이미지 + 텍스트 최종 합성
  - 비동기 처리: 작업 시작 시 task_id 반환, 폴링(상태 조회)을 통해 완료시 결과 제공
